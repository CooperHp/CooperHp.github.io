Git及其常用命令

发表于 2021-05-25 分类于 技术

官方文档、官方Book（中文）

典型工作流程

工作内容：测试修改不同的程序功能并发布。

  1. 指针位于主分支master
  2. 为修改程序功能创建并切换至新分支：git checkout -b dev
  3. 修改程序并提交：git add *；git commit -m "dev"
  4. 切换至主分支：git checkout master
  5. 将程序功能1的分支合并到主分支：git merge dev
  6. 删除程序功能1分支：git branch -d dev

分支

HEAD指针指向当前所在的分支，分支之间可以提交不同的版本，然后将不同分支进行合并。切换分支时，目录和文件会重置工作，最好提交当前分支的修改（或暂存stashing和修补提交commit amending）。

合并分支时冲突

当两个不同分支中对同一部分进行了不同修改时，在合并他们的时候就会产生合并冲突，git会暂停等待你解决合并产生的冲突。可以在合并冲突后的任意时刻使用git status查看因包含合并冲突而处于未合并unmerged状态的文件。git会在有冲突的文件中加入标准的冲突解决标记，在打开这些包含冲突的文件后就可手动解决冲突。一旦暂存这些原本有冲突的文件，git就会将它们标记为冲突已解决。

可以使用图形化工具来解决冲突：git mergetool。

配置

git config命令来设置控制git外观和行为的配置变量，这些变量存储在三个位置：

  • /etc/gitconfig 文件：系统级配置文件
  • ~/.gitconfig、~/.config/git/config文件：用户级配置文件，可以通过--global选项让git读写此文件，使得配置对系统中所有仓库生效
  • git仓库中的.git/config文件：仓库级配置文件，只针对当前仓库

变基

使用rebase命令将提交到某一分支上的所有修改都移至另一个分支上。

远程仓库副本

通过git fetch/git pull命令获取远程仓库内容到本地，fetch并未合并到本地仓库，此时可使用git merge实现远程仓库副本与本地仓库的合并。而git pull直接拉取远程副本并合并至本地。
